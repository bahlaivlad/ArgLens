# Catalog reference

`catalog.json` is the source of truth for every API ArgLens can rank. This
document covers the entry schema, the parameter kinds, how the scoring
engine uses them, and how to add or hand-tune entries.

---

## Entry shape

```json
{
  "name": "VirtualAlloc",
  "module": "kernel32",
  "category": "Memory",
  "signature": [
    "LPVOID lpAddress",
    "SIZE_T dwSize",
    "DWORD flAllocationType",
    "DWORD flProtect"
  ],
  "params": [
    { "kind": "ptr", "nullGood": true },
    { "kind": "size" },
    { "kind": "flags", "flags": "allocationType",
      "exactGood": [4096, 8192, 12288] },
    { "kind": "flags", "flags": "pageProtect",
      "exactGood": [4, 32, 64] }
  ],
  "system": false,
  "manual": true
}
```

| field | type | what it does |
|---|---|---|
| `name` | string | Display name. Ends with `A/W` when both ANSI and Unicode variants exist (expanded to `FooA` and `FooW` in the UI). |
| `module` | string | The DLL the symbol is exported from (`kernel32`, `advapi32`, `ntdll`, `wininet`, etc.). Used for grouping and for the MSDN URL fallback. |
| `category` | string | Free-form label shown in the candidate header — Memory, Files, Process, Threads, Sync, Loader, Registry, Services, Security, Sync, Network, Crypto, Handles, Native, System, Misc, Debug. |
| `signature` | string[] | One entry per parameter, **type and name together**, e.g. `"HANDLE hProcess"`. The IDA-compatible prototype is rendered from this. `signature.length` **must** match `params.length`. |
| `params` | object[] | Per-parameter scoring spec — see the kinds below. |
| `system` | bool | `true` for `ntdll` / native APIs; gets a small score bonus when other things tie. |
| `manual` | bool | Marks entries hand-tuned for ArgLens (vs. auto-generated by `tools/build-catalog.mjs`). The generator skips any name that already exists, so manual entries survive regenerations. |

---

## Parameter kinds

Each entry in `params` is a small object — `kind` plus optional flags that
relax or sharpen the scoring. The kinds are:

| kind | meaning | typical types |
|---|---|---|
| `ptr` | Generic pointer (in-pointer). Rejects NULL unless `nullGood`, rejects negative immediates. | `LPVOID`, `PVOID`, `LPSECURITY_ATTRIBUTES`, struct pointers |
| `stringptr` | Pointer to a string literal. Same shape rules as `ptr` but the prototype uses string types in `A/W` rendering. | `LPCSTR`, `LPCWSTR`, `LPCTSTR` |
| `codeptr` | Pointer to executable code — start routine, callback. | `LPTHREAD_START_ROUTINE`, `PAPCFUNC` |
| `outptr` | Out-pointer — caller passes the address of something that will be written. | `LPDWORD`, `PHANDLE`, `OUT` buffers |
| `handle` | Kernel/object handle. Rejects negative values unless `invalidHandleGood`. | `HANDLE`, `HMODULE`, `HKEY`, `SC_HANDLE` |
| `integer` | Plain integer — won't reject hex/decimal values, mild bonus. | `DWORD nIndex`, `UINT uExitCode` |
| `size` | Size/length value. Rejects 0 unless `zeroGood`; rejects negatives. | `SIZE_T`, `DWORD nBufferSize` |
| `flags` | Bitmask. Pair with `flags: "<groupName>"` (see below) to enable bit decoding. Rejects values with unknown bits unless `allowUnknownBits`. | `DWORD dwFlags`, `ACCESS_MASK` |
| `enum` | One of a small set of accepted values. Pair with `flags: "<groupName>"` for the value set. Rejects values not in the set. | `dwCreationDisposition`, `dwMoveMethod` |
| `bool` | Strict 0 or 1 only. | `BOOL bInherit` |

### Per-kind options

```ts
{
  kind: "ptr",
  nullGood: true,           // NULL is an accepted value
}

{
  kind: "handle",
  nullGood: true,           // hTemplateFile, hToken — NULL is fine
  invalidHandleGood: true,  // -1 is fine too (INVALID_HANDLE_VALUE)
}

{
  kind: "flags",
  flags: "pageProtect",     // see FLAG_DEFS groups below
  exactGood: [0x04, 0x20, 0x40],  // these exact values get a big bonus
  zeroGood: true,           // 0 is an accepted value (some flag args allow 0)
  allowUnknownBits: true,   // don't reject if extra bits are set
}

{
  kind: "enum",
  flags: "creationDisposition",   // 1..5 = CREATE_NEW..TRUNCATE_EXISTING
  exactGood: [3],                 // OPEN_EXISTING is the most common
}

{
  kind: "size",
  zeroGood: true,           // SIZE_T 0 is a valid free / unmap
}
```

### `FLAG_DEFS` group names

Defined at the top of [app.js](../app.js#L1). The current groups:

```
allocationType    pageProtect           fileAccess
fileShare         creationDisposition   fileAttributes
processAccess     threadAccess          wait
loadLibraryFlags  moveMethod            showWindow
socketAf          socketType            socketProtocol
ntCreateDisposition                     ntCreateOptions
sectionAccess     registryAccess        registryType
tokenAccess       serviceAccess         scmAccess
```

Add a new group by appending to `FLAG_DEFS` in [app.js](../app.js) — keys
are numeric values, values are display names.

## Scoring weights

Per-parameter ceiling values used by `maxScoreForParam`
([app.js](../app.js#L787)):

| kind | best-case score | conditions |
|---|---:|---|
| `flags` | 34 | with `exactGood` |
| `flags` | 24 | with a `flags` group only |
| `flags` |  4 | no group |
| `enum` | 34 | with `exactGood` |
| `enum` | 30 | with a group only |
| `bool` | 18 | |
| `ptr` / `stringptr` / `codeptr` / `outptr` | 18 | with `nullGood` |
| `ptr` / `stringptr` / `codeptr` / `outptr` | 16 | otherwise |
| `handle` | 18 | with `nullGood` |
| `handle` | 10 | otherwise |
| `size` | 12 | |
| `integer` |  7 | |

Each entry also starts with a `+44` base bonus for matching argument count.
Plus `+2` if `system: true`.

A neutral arg (unknown) contributes nothing to numerator OR denominator,
so it doesn't drag the percentage down. The final shown number is
`score / maxScore * 100`, clamped to `[0, 100]`.

## Adding a new entry by hand

Open `catalog.json`, add a JSON object somewhere in the array (order does
not matter — the app sorts itself). Example: adding `CreateFileMappingNumaA`
(currently only `CreateFileMappingNumaW` is present):

```json
{
  "name": "CreateFileMappingNumaA/W",
  "module": "kernel32",
  "category": "Memory",
  "signature": [
    "HANDLE hFile",
    "LPSECURITY_ATTRIBUTES lpFileMappingAttributes",
    "DWORD flProtect",
    "DWORD dwMaximumSizeHigh",
    "DWORD dwMaximumSizeLow",
    "LPCTSTR lpName",
    "DWORD nndPreferred"
  ],
  "params": [
    { "kind": "handle", "invalidHandleGood": true },
    { "kind": "ptr", "nullGood": true },
    { "kind": "flags", "flags": "pageProtect",
      "exactGood": [4, 32, 64] },
    { "kind": "integer", "zeroGood": true },
    { "kind": "size" },
    { "kind": "stringptr", "nullGood": true },
    { "kind": "integer" }
  ],
  "system": false,
  "manual": true
}
```

Set `manual: true` if you want the auto-generators to leave it alone.

That's it — no rebuild, the app picks it up next time you reload.

## Auto-generation

Two scripts under `tools/`:

- **`tools/build-catalog.mjs`** — scrapes Microsoft Learn for every header
  under `_base` (and `fileapi.h` + `processthreadsapi.h`), parses the C
  prototypes, classifies each parameter into the kinds above, and merges
  the result into `catalog.json`. Manual entries (`manual: true`) are
  preserved — the script only adds names that don't already exist.
- **`tools/add-nt-apis.mjs`** — adds the curated list of injection /
  anti-debug / ALPC Nt APIs that Microsoft doesn't fully document, then
  mirrors every `Nt*` entry as a `Zw*` alias.

Both are idempotent.

---

## Limitations to know about

- Auto-generated entries don't have `flags: "<group>"` set on their
  `flags`-kind params — the generator has no way to know that
  `DWORD dwAllocationType` maps to the `allocationType` group. Without
  it the param scores up to **+24** (vs. **+34** with `exactGood`). The
  243 hand-tuned `manual: true` entries do have these annotations and
  consequently rank more precisely.
- A/W deduping only happens within a single header in `build-catalog`,
  so a function whose A and W variants live in different headers ends
  up as two separate catalog entries.
- The catalog deliberately rejects 0-argument functions
  (`GetCurrentProcess`, `GetTickCount`, `WSACleanup`, …) — they aren't
  useful when the whole premise of the tool is "I see N arguments,
  what is this?".
